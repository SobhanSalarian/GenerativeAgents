### 2b. `reverie/backend_server/scenarios/information_consensus.py` (comments #4, #5)

Four changes made:

**Change 1 — Parameterised constructor**
`TRUE_OPTION` and `SIGNAL_COUNTS` moved from class-level constants to
constructor parameters with defaults that preserve primary dataset behaviour:
```python
def __init__(self, true_option="A", signal_counts=None):
    self.TRUE_OPTION   = true_option.upper()
    self.SIGNAL_COUNTS = signal_counts or {"A": 4, "B": 2, "C": 2}
    ...
```

**Change 2 — Fixed `_init_kwargs()`**
Was returning `{}`, meaning `clone_scenario()` in `experiment_runner.py`
would silently use default constructor for every trial after the first —
losing rotation settings in multi-rotation launchers.
```python
def _init_kwargs(self):
    return {"true_option": self.TRUE_OPTION, "signal_counts": self.SIGNAL_COUNTS}
```

**Change 3 — Fixed `_deterministic_reflection`**
Was a `@staticmethod` with hardcoded `"A"`. Converted to instance method;
replaced hardcoded string with `self.TRUE_OPTION`.

**Change 4 — Removed TRUE_OPTION flag from `group_status` memory string**
The `_record_scenario_memories` method was appending
`"X% of the group supported Option A (the majority-signal option)."` to the
memory string, giving agents a direct label for the correct answer.
The entire clause was removed; memory now records only the raw tally:
```
"At step N, the vote tally was: A:X, B:Y, C:Z."
```
Rationale: the public tally in `_consensus_context` is already visible to
ALL agents each step (including baseline). The memory flag was providing a
redundant hint that short-circuited genuine aggregation. Memory's real
mechanism is **temporal anchoring** (self_status across rounds), not
aggregation scaffolding. The 3:3:2 experiment (below) isolates this.

## 4. Key design insights recorded

### Why 3:3:2 + flag removal must be done together

The public tally (in `_consensus_context`) is shown to **all** agents every
step, including baseline. With 4:2:2, the tally shows A:4 from step 0 — a
clear plurality visible without any memory. Memory's role is therefore NOT
aggregation (the tally already reveals the answer); it is **temporal
anchoring** — `self_status` prevents individual agents from drifting across
rounds (cascade prevention, not aggregation per se).

With 3:3:2, the tally shows A:3=B:3 (dead tie). Now the only way to identify
A as correct is to weight evidence across rounds as signals accumulate. This
is genuine aggregation. The flag removal ensures memory does not re-introduce
a direct correct-answer hint. Both changes are necessary together.

### P1 mechanism

Static reflection injection bypasses the LLM `reflect()` call entirely —
the dominant CD saturation string is injected verbatim into
`scenario_reflections` before `_build_cognitive_context_for_scenario` runs.
No Anthropic API key required. The `memory_planning` arm (added in this
session) provides the counterfactual within the same experiment, same seed,
same personas.

## 6. Reflection condition clarification (methodological note for thesis)

### `full` vs `full_llm_reflection` — what each is

| | `full` | `full_llm_reflection` |
|---|---|---|
| Memory | ✅ | ✅ |
| Planning | ✅ | ✅ |
| Reflection | ✅ deterministic rule-based | ✅ LLM-synthesised (Park et al. style) |
| API calls/trial | ~1,195 | ~2,147 (~80% more) |
| Tokens/trial | ~832K | ~1.3M (~57% more) |

`full_llm_reflection` is the **faithful Park et al. (2023) implementation** — reflection
is a genuine LLM synthesis call over recent memories. `full` (deterministic) replaces
that call with a rule-based template, making it a deliberate simplification/ablation.

### Why `full` is not a flaw — it enables H6 and P1

The deterministic condition is intentional:
- **H6**: Reflection saturation is directly measurable (3 unique strings, top 2 = 95.8%
  of all injections). With LLM-generated reflection, semantic clustering would be needed.
- **P1**: The static injection experiment isolates whether the *content* of the
  deterministic string drives coordination. This test only makes sense because `full`
  uses a fixed, identifiable string.

The thesis argument: deterministic and LLM reflection both achieve coordination (70%
vs 100%), and P1 confirms *why* — the content converges to the same focal-point norm
either way.

### Why `full_llm_reflection` is not needed in the new scripts

| Script | Question being tested | `full_llm_reflection` needed? |
|--------|----------------------|-------------------------------|
| `run_p1.py` | Does reflection *content* drive coordination? | No — P1 tests the deterministic string specifically |
| `run_ic_counterbalance.py` | Does Option A label salience explain IC results? | No — reflection type is not the variable |
| `run_ic_noflag_332.py` | Does memory enable genuine aggregation (3:3:2)? | No — testing memory isolation, not reflection |
| `run_persona_panels.py` | Are CD findings robust across persona panels? | No — `full_llm_reflection` already showed 100% in primary data; expected to generalise |

### Thesis transparency requirement

Must explicitly state in §3 Methodology:
> "`full` uses a deterministic rule-based reflection rather than LLM-generated
> synthesis and is therefore not a verbatim replication of Park et al. (2023).
> `full_llm_reflection` is the faithful replication. The deterministic condition
> is retained as a deliberate ablation to enable H6 saturation measurement and
> the P1 confirmatory experiment."
